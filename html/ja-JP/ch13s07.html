13.7. 共同作業

13.7.1. グループで行う作業: groupware

グループウェアツールの保守は比較的複雑になる傾向があります。なぜなら、グループウェアツールは復数のツールを統合したもので、時には統合されたディストリビューションとの間の調整が難しい事を要求するからです。このため、沢山のグループウェアが過去に Debian から配布されていたにも関わらず、メンテナの不足か Debian に含まれる他の (新しい) ソフトウェアとの互換性の不足が原因で配布されなくりました。PHPGroupware、eGroupware、Kolab がこの例です。
→ http://www.phpgroupware.org/
→ http://www.egroupware.org/
→ http://www.kolab.org/
そうは言っても、すべてのグループウェアツールが Debian から配布されていないわけではありません。伝統的に「グループウェア」ソフトウェアが提供していた多くの機能は「標準的な」ソフトウェアに統合されつつあります。この統合により、グループウェアソフトウェア固有の特殊な要求が減りました。その一方で、グループウェアソフトウェアには専用のサーバが必要です。このような専用サーバ型グループウェアの良い例が Kolab です。Kolab は KDE (Kontact、Kmail など) 、Horde ウェブメール、Thunderbird (プラグイン経由で)、Microsoft Outlook に統合する事が可能です。より興味深いこととして、Citadel (citadel-suite パッケージに含まれます) と Sogo (sogo パッケージに含まれます) は代替品で、Debian Wheezy から配布されています。

13.7.2. インスタントメッセージシステム

社内用のインスタントメッセージシステムをセットアップする場合、Jabber が無難です。Jabber プロトコル (XMPP) はオープン標準で、機能には事欠きません。メッセージを暗号化する事が可能で、この機能はこの上なく最高の要素になる場合があります。さらに Jabber サーバと他のインスタントメッセージネットワークである ICQ、AIM、Yahoo、MSN などとの間にゲートウェイをセットアップする事が可能です。

ALTERNATIVE インターネットリレーチャット

Jabber の代わりに IRC を検討する事も可能です。IRC システムはチャンネルの概念を中心としています。チャンネルの名前はハッシュ記号 # で始まります。通常各チャンネルには特定の話題が設定されており、多くの人々がチャンネルに参加してその話題について議論する事が可能です (必要ならば、ユーザ同士が一対一でプライベートに会話する事も可能です)。IRC プロトコルは古く、メッセージをエンドツーエンドで暗号化する事が不可能です; IRC プロトコルを SSL の中にトンネリングすれば、ユーザとサーバ間の通信を暗号化する事も可能です。
IRC クライアントはより複雑で、集団環境の中における使用制限を行使するための多くの機能を備えています。例えば、チャンネルの「オペレータ」は、通常の議論が妨げられた場合、他のユーザをチャンネルから追い出したり、永久的に参加を禁止したりする責任を負うユーザです。
IRC プロトコルはとても古いため、多くのユーザグループの需要に応える多くの IRC クライアントが利用可能です; IRC クライアントには XChat、Smuxi (GTK+ に基づくグラフィカルクライアント)、Irssi (テキストモード)、Erc (Emacs への統合)、Chatzilla (Mozilla ソフトウェアスイートに含まれます)、などがあります。

QUICK LOOK Ekiga を使ったビデオ会議

Ekiga (旧名 GnomeMeeting) は Linux ビデオ会議用の最も有名なアプリケーションです。Ekiga は安定で機能的な特徴を持ち、ローカルネットワーク上でとても簡単に使う事が可能です。使われているファイヤーウォールが H323 や SIP テレビ会議プロトコルの全機能を明示的にサポートしていない場合、グローバルネットワーク上でサービスを設定することはかなり複雑になります。
ファイヤーウォールの背後で実行されている Ekiga クライアントが 1 台だけの場合、その設定はかなり分かりやすいものです。専用のホストに幾つかのポートを転送する設定を行うだけです: TCP ポート 1720 番 (入って来る接続をリッスン)、TCP ポート 5060 番 (SIP 用)、TCP ポート 30000 番から 30010 番 (開かれた接続の制御用)、UDP ポート 5000 番から 5013 番 (音声と動画データの伝送と H323 プロキシ登録用) です。
ファイヤーウォールの背後で復数の Ekiga クライアントを実行する場合、設定は著しく複雑なものになります。H323 プロキシ (例えば gnugk パッケージ) をセットアップする必要がありますし、H323 プロキシの設定はとても難しいものです。

13.7.2.1. サーバの設定

Jabber サーバをセットアップすることはかなり分かりやすいものです。ejabberd パッケージをインストールした後、dpkg-reconfigure ejabberd を実行することで、デフォルトのドメインをカスタマイズしたり、管理者アカウントを作成する事が可能です。Jabber サーバはそのサーバを指す有効な DNS 名が必要という点に注意してください。ネットワーク管理者は事前に DNS を設定しておく必要がある場合があります。Falcot Corp の管理者は Jabber サーバを担当するマシンに jabber.falcot.com を割り当てました。
最初のセットアップが終了したら、http://jabber.falcot.com:5280/admin/ から利用可能にされているウェブインターフェース経由で Jabber サービス設定を制御する事が可能です。ここで要求されるユーザ名とパスワードは初期設定中に設定したものを使います。ユーザ名は設定したドメインで修飾されたものを使う点に注意してください: つまり admin アカウントは admin@jabber.falcot.com のように使います。
ウェブインターフェースを使えば、設定ファイルを編集する必要が無くなりますが、すべての作業が簡単になるわけではありません。なぜなら、多くのオプションは特殊な構文を使って設定する必要があり、その構文を知らなければ設定できないからです。この理由から /usr/share/doc/ejabberd/guide.html を読む事をおすすめします。

13.7.2.2. Jabber クライアント

GNOME は Empathy (よく似た名前のパッケージに含まれます) を提供します。Empathy は最小主義的クライアントで、デスクトップの通知エリア (デフォルトの GNOME 設定の場合は右上隅) に統合されます。Empathy は Jabber だけでなく多くのインスタントメッセージプロトコルをサポートします。
KDE は Kopete (同名のパッケージに含まれます) を提供します。

13.7.3. FusionForge を使った共同作業

FusionForge はフリーソフトウェアプロジェクト用のホスティングサービスである SourceForge を起源とする共同開発ツールです。FusionForge はフリーソフトウェアの標準的な開発モデルに基づく総合的な取り組み方を採用しています。SourceForge のコードが商標で守られるようになった後にも、FusionForge 自体は進化し続けました。SourceForge コードの最初の作者である VA Software はこれ以上フリー版を公開しない事を決定しました。最初のフォークである (GForge) が同じ道をたどった時も同じ事が起きました。多くの人と組織が FusionForge の開発に参加し続けたため、現在の FusionForge には開発に対する古典的な姿勢を採用する機能および、純粋なソフトウェア開発に該当しないプロジェクトが含まれます。
FusionForge はプロジェクトを管理、追跡、調整するための復数のツールの合併とみなす事が可能です。これらのツールはおおまかに言って 3 種類に分類する事が可能です:
  • 連絡ツール: ウェブフォーラム、メーリングリスト管理、プロジェクトからのニュースを発表するための告知システム;
  • 追跡ツール: 進捗を制御したりタスクの予定を立てるためのタスクトラッカー、バグトラッカー (またはパッチ、機能要求、「チケット」と呼ばれるその他の要素、調査;
  • 共有ツール: プロジェクトに関連する文書の唯一の原本を提供するための文書マネージャ、包括的なファイルリリースマネージャ、プロジェクト専用のウェブサイト。
FusionForge は様々な開発プロジェクト対象にしているため、FusionForge はソースコード管理を担う CVS、Subversion、Git、Bazaar、Darcs、Mercurial、Arch、などの多くのツールを統合しています。ソースコード管理は「設定管理」や「バージョン管理」などと呼ばれる場合があります。ソースコード管理を担うプログラムは登録された全ファイル (多くの場合ソースコードファイル) の履歴をファイルに対して行われたすべての変更と一緒に保存したり、復数の開発者がプロジェクトの同じ部分に対して同時に仕事を行った場合にその変更をマージする事が可能です。
ソースコード管理ツールのほとんどは、きめ細かく調整されたパーミッションシステムと電子メールを使った幾つかのイベントの通知システムを備えたウェブインターフェースを介して利用したり管理する事が可能です。
残念なことに、FusionForge は Wheezy がフリーズされた時に流動的な状態でした。そのため、標準的な Wheezy に含まれません; 執筆時点で、バックポートはまだ利用可能ではありませんが、すぐに利用可能になる事が予想されます。
