7.2. 常套手段

この節の目的は、管理者が頻繁に実行する特定の操作に関する、幾つかの一般的な秘訣を示すことです。もちろんこれらの手順は包括的にすべての考え得る場合をカバーするものではありませんが、より困難な状況に直面した際の出発点として機能します。

DISCOVERY フランス語の文書

多くの場合、非英語に翻訳された文書は対応するパッケージ名の後ろに -lang を付けた (ここで lang は 2 文字の ISO 言語コードです) 名前の別パッケージとして提供されています。
たとえば、apt-howto-fr パッケージには APT に関する HOWTO のフランス語翻訳が含まれています。同様に、quick-reference-fr と debian-reference-fr パッケージは Debian に関するリファレンスガイド (最初 Osamu Aoki さんが英語で書いた文書) のフランス語版です。

7.2.1. プログラムの設定

未知のパッケージを設定する場合、段階を追って進まなければいけません。最初に、パッケージメンテナの書いた文書を読んでください。/usr/share/doc/package/README.Debian を読むことで、そのソフトウェアを簡単に使うための詳細な規定を学ぶことが可能です。このファイルには、プログラムの元の挙動 (HOWTO などの一般的な文書に書かれている挙動) からの違いが書かれているため、必ず読まなければいけない場合があります。またこのファイルには、ありふれた問題で時間を無駄にするのを避けるために、最も一般的なエラーの詳細も書かれています。
その後、ソフトウェアの公式文書を確認して下さい - 既存の様々な文書源を見つける方法は前の節を参照してください。dpkg -L package コマンドでパッケージに含まれるファイルのリストが表示されます; このコマンドを使えば、利用可能な文書 (と同時に /etc/ に格納された設定ファイル) を素早く特定することが可能です。dpkg -s package でパッケージのメタ情報が表示されるので、そこから推奨パッケージや提案パッケージが分かります; そこから、ソフトウェアの設定を簡単に行うための文書またはユーティリティを見つけることが可能です。
最後に、設定ファイルには通常、各設定項目に関して設定可能な値をそれぞれ詳細に説明した説明用コメントが数多く書かれています。このため、利用可能な行を選んで有効化するだけで十分な場合もあります。また、設定ファイルの例が /usr/share/doc/package/examples/ ディレクトリの中に格納されている場合もあります。設定ファイルの例は、自分の設定ファイルのたたき台になります。

DEBIAN POLICY 例の場所

すべての例は必ず /usr/share/doc/package/examples/ ディレクトリにインストールされます。このディレクトリに格納できる例は、設定ファイル、プログラムのソースコード (ライブラリの使用例)、ある場合に管理者が使うデータ変換スクリプトなどです。例が特定のアーキテクチャに固有なものの場合、例は /usr/lib/package/examples/ にインストールされ、このディレクトリには /usr/share/doc/package/examples/ ディレクトリに含まれるファイルを指すリンクが含まれるべきです。

7.2.2. デーモンの挙動を監視する

デーモンの挙動を理解するのはやや面倒です。なぜなら、デーモンは管理者と直接的に対話するものではないからです。デーモンが実際どのような仕事をしているかを確認するには、デーモンをテストする必要があります。たとえば、Apache (ウェブサーバ) デーモンの挙動を確認するには、HTTP リクエストを使ってテストします。
テストを行うために、デーモンは通常、自分のしたことだけでなく発生したエラーも含めてすべてを、「ログファイル」や「システムログ」と呼ばれるログに記録します。ログは /var/log/ の中か、このディレクトリのサブディレクトリの中に保存されます。それぞれのデーモンの正確なログファイル名を知るには、文書を確認して下さい。注意: そのテストによって考え得る使用例のすべてが満足されない限り、テストは必ずしも 1 回で十分とは限りません; 幾つかの問題は特定の状況下でしか起こりません。

TOOL rsyslogd デーモン

rsyslogd は特別です: このデーモンは他のプログラムが送信したログ (内部的システムメッセージ) を収集します。それぞれのログエントリはサブシステム (電子メール、カーネル、認証など) と優先度が付けられています; rsyslogd はこれら 2 つの情報を使って各エントリをどのように処理するかを決めます。ログメッセージは、複数のログファイルに記録される場合もあれば、管理者のコンソールに送信される場合もあります。設定の詳細は /etc/rsyslog.conf 設定ファイル (同名のマニュアルページで説明されています) で定義されています。
C 言語にはログ送信専用の関数があるので、これを使うと rsyslogd デーモンを簡単に使えるようになります。しかしながら、幾つかのデーモンは自分でログファイルを管理します (たとえば、samba、Linux 上で動く Windows 共有の実装、がその場合に当てはまります)。

BACK TO BASICS デーモン

デーモンはユーザが明示的に呼び出すプログラムではなく、バックグラウンドで実行され、特定の状態が満足されたら作業を行うプログラムです。多くのサーバプログラムはデーモンです。専門用語の名前の最後に付けられた「d」という文字は通常それがデーモンであることを意味しています (sshd、smtpd、httpd、など)。
予防手段として、管理者は日常的に最重要なサーバログを読むべきです。そうすれば、不満を抱くユーザが報告する前に問題を突き止めることができます。実際のところ、ユーザは報告の前に、問題が数日間にわたって繰り返して起こるまで待つかもしれません。多くの場合、長いログファイルの内容を分析する特定のツールが有ります。特に、ウェブサーバ (Apache 向けの analog、awstats、webalizer)、FTP サーバ、プロキシ/キャッシュサーバ、ファイヤーウォール、電子メールサーバ、DNS サーバ、プリンタサーバについてはそのログを解析するユーティリティが存在します。幾つかのユーティリティはモジュール式で機能し、複数のログファイルを解析できます。lire や modlogan がこれに相当します。他のツール、たとえばlogcheck (14章セキュリティで紹介するソフトウェア) など、はログファイルから対処が必要な警告を検索します。

7.2.3. メーリングリストで助けを求める

色々検索しても問題の原因が判らなかった場合、他の人、より経験豊富な人、から助けを受けることも可能です。debian-user@lists.debian.org メーリングリストはこのために用意されています。他のコミュニティと同様、このメーリングリストにも従う必要のあるルールがあります。質問の前に、メーリングリストに最近投稿された議論および公式文書の中で、自分の問題がまだ報告されていない点を確認するべきです。
→ http://wiki.debian.org/DebianMailingLists
→ http://lists.debian.org/debian-user/

TIP ウェブでメーリングリストを読む

debian-user@lists.debian.org などの流量の多いメーリングリストは、ディスカッションフォーラム (またはニュースグループ) 的に読むと良いかもしれません。Gmane.org を使うと、この形式で Debian メーリングリストを参照することが可能です。上に挙げたメーリングリストは以下から閲覧できます:
→ http://dir.gmane.org/gmane.linux.debian.user

BACK TO BASICS ネチケット

一般的に言って、電子メーリングリスト上でのやりとりはすべてネチケットを守るべきです。ネチケットとは、礼儀作法から避けるべき間違いまで、常識的なルールを指しています。
→ http://tools.ietf.org/html/rfc1855
以下の 2 条件を満足した時点で、問題をメーリングリストに投稿してみてください。関連する情報をできる限り含めること: 実施した様々なテスト、参照した文書、その問題を診断する際に使った方法、関連するか関連する可能性のあるパッケージ、など。Debian バグ追跡システム (BTS、傍注TOOL バグ追跡システム参照) で同様の問題がないか確認し、検索の結果を挙げ、見つかったバグへのリンクを提供すること。以下から BTS を使い始めることが可能です:
→ http://www.debian.org/Bugs/index.html
礼儀正しく正確であればあるほど、答えを得られる (少なくとも多少の反応を得られる) チャンスが大きくなります。私信で関連する情報を受け取った場合、その情報を要約して公にすることを考えてください。そうすれば、他の人がその恩恵を受けられます。またこうすることで、様々な検索エンジンを通じて、同じ疑問を持つ人がメーリングリストのアーカイブからその解決策を得ることが可能です。

7.2.4. 問題が難しすぎる場合のバグ報告

問題を解決するために行ったすべての努力が失敗に終わった場合、問題の原因はあなたの責任の及ぶところになく、プログラムのバグかもしれません。この場合、適切な手続きは Debian にバグを報告するか、上流開発者に直接報告するかです。これを行うには、可能な限り問題を分割し、問題を再現できる最小のテスト環境を作ってください。問題の原因が明らかにプログラムにあるとわかっている場合、dpkg -S file_in_question コマンドで関連するパッケージを見つけることが可能です。バグ追跡システム (http://bugs.debian.org/package) を使って、そのバグがまだ報告されていないことを確かめてください。その後、reportbug コマンドでバグ報告を送信してください。その際には、可能な限り多くの情報を含め、特に誰もがバグを再現できる最小テストケースの完全な説明を含めてください。
この章では、効率的な問題解決の手段を解説しました。以降の章ではこの手段を使うことになるでしょう。必要に応じて何度でも使ってください!
